# ТЗ: лимит смен никнейма в профиле (Supabase / Postgres)

**Версия:** 1  
**Контекст:** таблица `public.profiles`, клиенты — мобильное приложение (Flutter) через PostgREST.

---

## 1. Бизнес-логика (вариант B)

| Правило | Описание |
|--------|----------|
| Первый ник | Если раньше `username` был `NULL`, первое сохранение ника **не увеличивает** счётчик смен. |
| Смена ника | Если старый ник **уже был** (после нормализации — см. ниже) и новое значение **отличается** — это **одна смена**, счётчик **+1**. |
| Окно | В одном «окне» допускается не более **4** таких смен. |
| Пауза | После **4-й** смены в текущем окне выставляется **блокировка на 7 суток** от момента этой смены. |
| После паузы | Когда текущее время **≥** дедлайна блокировки, при **любом** `UPDATE` строки профиля счётчик **обнуляется**, дата блокировки **сбрасывается** — открывается **новое окно** из 4 смен. |
| Уникальность | Никнейм **уникален** среди непустых значений (регистр не важен): уже описано отдельным unique-индексом на `lower(username)`. |

**Нормализация ника для сравнения «старый / новый»:** как в существующем триггере: `lower(trim(username))`. Триггер лимита должен выполняться **после** триггера нормализации ника (порядок имён триггеров в Postgres).

---

## 2. Данные в `profiles`

| Поле | Тип | Обязательность | Смысл |
|------|-----|----------------|--------|
| `username` | `text` | опционально | Текущий ник; пустое/`NULL` — ника нет. |
| `username_change_count` | `int` | `NOT NULL`, default `0` | Сколько смен выполнено **в текущем окне** (0…4). Клиент **не должен** доверять своему значению при записи — пересчитывает сервер. |
| `username_next_change_allowed_at` | `timestamptz` | `NULL` или дата | Пока `now() <` этого времени — **нельзя** менять ник с непустого старого значения (активна пауза после 4 смен). `NULL` — паузы нет. |

**Ограничение:** `CHECK (username_change_count >= 0 AND username_change_count <= 4)`.

**Политика записи:** клиент в `POST/PATCH` **не передаёт** `username_change_count` и `username_next_change_allowed_at` как источник истины (или передача игнорируется триггером). Итоговые значения выставляет **триггер `BEFORE UPDATE`**.

---

## 3. Поведение триггера (кратко)

1. Взять `eff_count` и `cool_until` из **старой** строки.
2. Если `cool_until IS NOT NULL` и `now() >= cool_until` → `eff_count := 0`, `cool_until := NULL` (сброс окна).
3. Если меняется ник (`old_u` и `new_u` различаются после `lower(trim)`) и **старый ник не `NULL`**:
   - если `eff_count >= 4` и `cool_until IS NOT NULL` и `now() < cool_until` → **ошибка** `username_change_cooldown` → **P0002**; в `HINT` — ISO-время дедлайна (UTC), удобно для UI;
   - если `eff_count >= 4` (защита от неконсистентных данных) → **ошибка** `username_change_limit_reached` → **P0001**;
   - иначе `eff_count := eff_count + 1`;
   - если после инкремента `eff_count >= 4` → `cool_until := now() + interval '7 days'`.
4. Записать в `NEW` поля `username_change_count` и `username_next_change_allowed_at` из вычисленных `eff_count` и `cool_until`.

---

## 4. Коды ошибок для клиента / API

| Ситуация | Текст / сообщение | `ERRCODE` (Postgres) | Примечание |
|----------|-------------------|----------------------|------------|
| Пауза после 4 смен, смена ника с непустого | `username_change_cooldown` | `P0002` | В `HINT` — ISO UTC дедлайна |
| Неконсистентное состояние (например, 4 без даты) | `username_change_limit_reached` | `P0001` | Редкий кейс |
| Дубликат ника | unique violation | `23505` | Существующий unique index |

Клиент маппит в понятные сообщения пользователю.

---

## 5. RLS и права

- Должна сохраняться политика: пользователь может **обновлять только свою** строку `profiles` (`id = auth.uid()`).
- **Select** на эти поля — по текущей политике (чтобы видеть счётчик и дату паузы в приложении).

## 7. Что не входит в это ТЗ

- **Вариант A** (rolling «4 смены за последние 7 суток» без явной паузы) — отдельное согласование и другая модель данных.
- Смена email, аватаров, остальных полей профиля — без изменения правил лимита ника, кроме триггера сброса окна по истечении `username_next_change_allowed_at` при любом `UPDATE`.
