# ТЗ: справочники Country / City / Category (Supabase)

## 1. Цель

Единый источник правды для списков **стран**, **городов** и **категорий профиля**. В профиле пользователя и в остальной БД хранятся только **коды** (строковые идентификаторы), не переводимые подписи. Переводы названий для UI — на клиенте (l10n / ARB по кодам).

## 2. Принципы

| Сущность   | В БД хранится                         | Уникальность                          |
|-----------|----------------------------------------|----------------------------------------|
| Страна    | `country_code` (ISO 3166-1 alpha-2)    | один код на страну                     |
| Город     | `country_code` + `city_code`           | уникальна **пара** `(country, city)`   |
| Категория | `category_code` (slug)                 | один код на категорию                  |

- Коды — **стабильные**: не меняются при смене языка UI; переименования в маркетинге делаются на фронте.
- Новые значения добавляются **миграцией / сидом** (или админкой позже), не «произвольной строкой» из клиента без валидации.

## 3. Схема таблиц (PostgreSQL, `public`)

### 3.1. `public.countries`

| Колонка        | Тип        | Ограничения     | Описание |
|----------------|------------|-----------------|----------|
| `code`         | `text`     | `PRIMARY KEY`   | ISO 3166-1 alpha-2, upper-case, длина 2 (`KZ`, `RU`) |
| `is_active`    | `boolean`  | `NOT NULL DEFAULT true` | Скрыть страну из выдачи без удаления строк |
| `sort_order`   | `int`      | `NOT NULL DEFAULT 0`    | Порядок в списках |

**Индексы:** по необходимости `WHERE is_active` (Partial index опционально).

### 3.2. `public.cities`

| Колонка         | Тип        | Ограничения |
|-----------------|------------|-------------|
| `country_code`  | `text`     | `NOT NULL REFERENCES public.countries(code) ON DELETE RESTRICT` |
| `city_code`     | `text`     | `NOT NULL`, формат: латиница/цифры/подчёркивание, lower-case или upper-case — **зафиксировать один стиль в проекте** |
| **PK**          |            | `PRIMARY KEY (country_code, city_code)` |
| `is_active`     | `boolean`  | `NOT NULL DEFAULT true` |
| `sort_order`    | `int`      | `NOT NULL DEFAULT 0` |

**Индексы:** уже покрыто PK; при необходимости `INDEX (country_code) WHERE is_active` для выборки по стране.

### 3.3. `public.profile_categories`

| Колонка         | Тип        | Ограничения |
|-----------------|------------|-------------|
| `code`          | `text`     | `PRIMARY KEY`, slug (`beauty`, `music`, …) |
| `is_active`     | `boolean`  | `NOT NULL DEFAULT true` |
| `sort_order`    | `int`      | `NOT NULL DEFAULT 0` |

**Имя таблицы** можно заменить на `categories`, если нет конфликтов; в ТЗ зафиксировано назначение — категории **профиля**.

## 4. Связь с профилем пользователя

Таблица `public.profiles` (или эквивалент):

- `country_code` → `text`, `FK → countries(code)` или проверка без FK, если исторически бывают «старые» коды (предпочтительно FK + миграция данных).
- `city_code` → `text`, **обязательно согласован с** `country_code`: либо составной FK через отдельную таблицу-декларацию, либо **триггер/check** на существование строки в `cities` для пары `(country_code, city_code)`.
- `category_id` в текущем коде приложения — **привести к единому имени**: либо переименовать колонку в `category_code`, либо оставить имя `category_id`, но тип и семантика = **код категории** (`text`, FK → `profile_categories(code)`). В ТЗ ниже используется семантика **код категории**.

Рекомендация: колонка `category_code` + `FK REFERENCES profile_categories(code)`.

## 5. Row Level Security (RLS)

Политики по умолчанию для MVP:

| Таблица               | `SELECT`                         | `INSERT/UPDATE/DELETE` |
|-----------------------|----------------------------------|-------------------------|
| `countries`           | `authenticated` (при необходимости и `anon`) | только `service_role` / отдельная роль админа |
| `cities`              | то же                            | то же                   |
| `profile_categories`  | то же                            | то же                   |

Точный набор ролей согласовать с продуктом (нужны ли справочники до логина — тогда `anon` на `SELECT`).

## 6. Контракт API для клиента (PostgREST)

Формат ответа — JSON PostgREST; клиент фильтрует `is_active = true`, сортирует по `sort_order`, затем подставляет **лейблы** из l10n по `code`.

### 6.1. Страны

```http
GET /rest/v1/countries?is_active=eq.true&select=code,sort_order&order=sort_order.asc
```

### 6.2. Города (по стране)

```http
GET /rest/v1/cities?country_code=eq.KZ&is_active=eq.true&select=country_code,city_code,sort_order&order=sort_order.asc
```

### 6.3. Категории профиля

```http
GET /rest/v1/profile_categories?is_active=eq.true&select=code,sort_order&order=sort_order.asc
```

При росте нагрузки — один **RPC** `get_reference_lists(country_code text default null)` с JSON-ответом; для MVP достаточно трёх запросов.

## 7. Валидация и кодстайл кодов

- **Страна:** ровно 2 символа A–Z (нормализовать при сиде в upper-case).
- **Город:** длина и charset зафиксировать (например `^[A-Z0-9_]{2,32}$` в upper-case); один стиль на весь проект.
- **Категория:** slug `^[a-z0-9_]{2,64}$` (lower-case).

Проверки: `CHECK` на таблицах или отдельные `DOMAIN` типы в PostgreSQL (опционально).

## 8. Сиды (начальное наполнение)

- Минимальный набор стран (например KZ, RU, …) — SQL-файл в репозитории или Supabase migration.
- Города — только релевантный список под продукт (не обязательно полный гео-каталог).
- Категории — согласованный с продуктом список `code` + `sort_order`.

Документировать соответствие **код → ключ l10n** на стороне Flutter (отдельная таблица в команде, не в БД).

## 9. Вне scope данного ТЗ

- Переводы названий в БД (i18n-таблицы) — не требуются при модели «либелы на фронте».
- Поиск городов по подстроке / геокодинг.
- Версионирование справочников и CDN-кэш — при необходимости отдельным ТЗ.

## 10. Критерии приёмки

- Три таблицы созданы, связи и PK соответствуют п. 3–4.
- RLS включён, политики чтения для клиентских ролей работают.
- Сиды применяются без ошибок; клиент может построить выпадающие списки по трем endpoint’ам п. 6.
- Профиль сохраняет только коды; попытка сохранить несуществующую пару город/страна отклоняется **FK/триггером** (по согласованию с командой).
