# ТЗ: Supabase — социальный граф, лента, состояние аккаунта

**Версия:** 1.1  
**Статус:** нормативный документ для реализации и приёмки бэкенда (Postgres + RPC + RLS + при необходимости Edge/Cron).  
**Границы:** Flutter и клиентский кэш описываются только там, где нужен **контракт ответа API** (инвалидация ленты — ответственность клиента при lazy-модели).

**Уклон v1.1:** триггеры на графе — **только O(1)** к двум строкам `profiles`; тяжёлая логика, уведомления и антифрод — **вне** триггерного тела (см. §3.5).

---

## 1. Цели и принципы

### 1.1 Цели

- Пользователь может **подписываться** и **отписываться** (направленное ребро `follower → following`).
- **Источник истины** по графу — таблица рёбер; `followers_count` / `following_count` в `profiles` — **кэш для UI**.
- **Лента по подпискам** строится из графа + контента (посты), с возможностью ранжирования.
- **Состояние аккаунта** (`active` / `hibernate` и см. п. 8) **не разрушает** граф при смене режимов.
- Система **идемпотентна**, **безопасна** (RLS + явные RPC), **наблюдаема** (drift счётчиков, ошибки RPC).

### 1.2 Архитектурные принципы (обязательны)

| Принцип | Требование |
|--------|-------------|
| Разделение | `profile_follows` ≠ `posts` / `media` ≠ `profiles.account_state` |
| Идемпотентность | Повторный `follow` не создаёт дубликат ребра и не дублирует side-effects |
| Консистентность кэша | Счётчики на профиле сверяются с графом (reconcile + алерт при drift) |
| Единая политика взаимодействия | Блокировки и «можно ли A действовать на B» — **одна** функция/контракт, переиспользуемая в follow / feed / (вне ТЗ) chat / mentions |
| **Триггеры = горячий путь** | Внутри триггера на `profile_follows` — **только** два `UPDATE profiles … ±1` по PK; без `COUNT`, без вторых таблиц, без сетевых вызовов |

## 3. Модель данных (Supabase / Postgres)

### 3.1 Таблица социального графа — `public.profile_follows`

**Назначение:** одно направленное ребро «A читает B».

| Колонка | Тип | Обязательно |
|--------|-----|-------------|
| `follower_id` | `uuid` → `profiles.id` | да |
| `following_id` | `uuid` → `profiles.id` | да |
| `created_at` | `timestamptz`, default `now()` | да |

**Ограничения:**

- `PRIMARY KEY (follower_id, following_id)`
- `CHECK (follower_id <> following_id)`
- `FOREIGN KEY` на `profiles` с **`ON DELETE CASCADE`** по обоим концам (целостность при удалении профиля; поведение при soft-delete профиля — отдельное решение в п. 8.3)

**Индексы:**

- `(follower_id)` — список «на кого подписан я»
- `(following_id)` — список «кто подписан на меня»

**Источник истины:** только эта таблица для факта подписки.

### 3.2 Кэш счётчиков — `public.profiles`

| Колонка | Семантика |
|--------|-----------|
| `followers_count` | Число рёбер, где `following_id = profiles.id` |
| `following_count` | Число рёбер, где `follower_id = profiles.id` |

**Правило:** значения **кэш**; истина — `profile_follows`. Обязательна процедура **reconcile** (п. 6).

### 3.3 Блокировки — `public.profile_blocks` (новая сущность)

**Назначение:** единая основа для «нельзя взаимодействовать».

Минимальный контракт (уточнить направленность в продукте и зафиксировать в миграции):

| Колонка | Тип | Примечание |
|--------|-----|------------|
| `blocker_id` | `uuid` | кто заблокировал |
| `blocked_id` | `uuid` | кого заблокировали |
| `created_at` | `timestamptz` | |

- `PRIMARY KEY (blocker_id, blocked_id)`
- `CHECK (blocker_id <> blocked_id)`
- Индексы под выборки «все блокировки пользователя»

**Семантика по умолчанию для follow:** если существует блок в любом из направлений, релевантных продукту (задать явно: только `blocker→blocked` или также обратная проверка), создание ребра **запрещено**.

### 3.4 Уведомления — `public.notification_events` (или эквивалент)

**Назначение:** запись события после успешного создания ребра (at least once + дедуп).

Минимальный контракт:

| Колонка | Смысл |
|--------|--------|
| `dedupe_key` | Уникальный ключ, напр. `follow:{follower_id}:{following_id}` или + `type` |
| `recipient_id` | Кому (обычно `following_id`) |
| `payload` | `jsonb` (тип, actor_id, текст для клиента) |
| `created_at` | |

- `UNIQUE (dedupe_key)` **или** уникальность по `(recipient_id, dedupe_key)` в зависимости от модели хранения
- Политика TTL дедупа (например 48h) — опционально, документировать

**Доставка** в push — вне ядра Postgres (Edge / очередь); ядро фиксирует **запись события** и **отсутствие дублей** по ключу.

### 3.5 Производительность и корректность: триггеры под большой нагрузкой

**Цель:** при миллионах пользователей и пиках follow/unfollow **не лагать** из‑за БД и оставаться **корректными** (счётчики и граф согласованы в пределах заявленной модели кэша).

#### 3.5.1 Что допустимо в триггере на `profile_follows`

| Разрешено | Запрещено |
|-----------|-----------|
| Два коротких `UPDATE public.profiles SET … = … ± 1 WHERE id = $pk` по **первичному ключу** `profiles.id` | `SELECT count(*)` по графу, сканы ленты, join к большим таблицам |
| `greatest(count - 1, 0)` на delete | `RAISE NOTICE`, логирование в сторонние таблицы на каждую строку |
| Тело функции **короткое**, без ветвлений к бизнес-правилам | `pg_notify`, HTTP (`http`), вызовы Edge из триггера |
| Язык: предпочтительно **`LANGUAGE sql`** для триггер-функции, если хватает одного `UPDATE`/`RETURNING` — меньше накладных расходов, чем `plpgsql` | Триггер `BEFORE` с тяжёлой валидацией — только если без альтернативы |

**Инвариант:** время работы триггера на одно ребро — **O(1)** относительно размера графа; сложность не должна расти с числом подписчиков у других пользователей.

#### 3.5.2 Почему два `UPDATE` по `profiles.id` — нормально

- Postgres находит строку профиля по **PK** за одно обращение к индексу / куче.
- Операции `+= 1` / `greatest(-1)` — атомарны на уровне строки; корректность при конкуренции обеспечивается блокировкой строки на время `UPDATE` (короткая блокировка).

**Узкое место (осознанно):** «звёздный» профиль B — при одновременном всплеске follow на B **много транзакций** сериализуются на **одной строке** `profiles` (поле `followers_count`). Это не баг триггера, это физика hot row. Митигации:

1. **Не раздувать** работу на этой строке ничего лишнего (см. запреты выше).  
2. **Reconcile** (п. 6) и **кэш в UI** снимают страх перед редким кратковременным отставанием при экстремальном QPS.  
3. При необходимости v2 продукта: **отложенный счётчик** (очередь «increment B») — только если метрики hot row станут bottleneck; в ТЗ v1 это **не обязательно**.

#### 3.5.3 Альтернатива без триггера (допустимый паттерн)

Единый путь: **`follow_user` / `unfollow_user`** в одной транзакции выполняют `INSERT`/`DELETE` в `profile_follows` и **сразу два `UPDATE profiles`** с тем же смыслом ±1.

- **Плюс:** вся логика в одном месте, проще рассуждать о транзакции.  
- **Минус:** любой другой код, который когда‑либо вставит ребро мимо RPC, **сломает** счётчики — поэтому нужен жёсткий **REVOKE INSERT** на таблицу для клиента.

**Требование ТЗ:** выбрать **один** подход на проект — «триггеры» **или** «только RPC», не дублировать оба (+1 дважды).

Текущая миграция `20260420120000_profile_follows_social_graph.sql` использует **триггеры**; они соответствуют §3.5.1 при условии, что тело остаётся минимальным.

#### 3.5.4 Индексы и вставка в `profile_follows`

- Уже заданы индексы по `follower_id` и `following_id` — достаточно для списков и для FK-проверок.  
- `PRIMARY KEY (follower_id, following_id)` даёт уникальность и быстрый `ON CONFLICT DO NOTHING` в RPC.

#### 3.5.5 Уведомления и тяжёлая логика — строго вне «толстого» триггера на граф

После успешного `follow_user`:

- Запись в `notification_events` с **дедуп-ключом** — в **той же транзакции**, что и `INSERT` ребра, **но** вызывается из **тела RPC** после вставки ребра, **не** из тяжёлого `AFTER INSERT` триггера на `profile_follows` (чтобы триггер на графе оставался холодным и предсказуемым по времени).

Исключение: отдельный минимальный триггер только на `INSERT INTO notification_events` одной строкой — допустимо; **не** смешивать обновление счётчиков, нотификации и сторонние вызовы в одном триггере.

#### 3.5.6 Корректность при сбоях

- Транзакция: ребро + счётчик + (опционально) нотификация — **все commit или все rollback**.  
- Reconcile (п. 6) — страховка от дрейфа после редких ошибок, ручных правок или будущих путей миграции данных.

## 4. Функции Postgres (RPC) — контракт

Все мутации графа — **только через `SECURITY DEFINER`** RPC с `SET row_security TO off` там, где это необходимо для согласованности, и с явным `REVOKE` прямого DML с клиента на `profile_follows` (если выбрана модель «только RPC»).

### 4.1 `public.follow_user(p_target uuid) RETURNS void`

**Предусловия:**

1. `auth.uid()` не null  
2. `p_target` существует в `profiles`  
3. `auth.uid() <> p_target`  
4. `can_user_interact(auth.uid(), p_target)` = true (п. 5)  
5. Цель **не** в режиме сна: `profiles.account_state <> 'hibernate'` (и при необходимости `content_visible` — по продукту)  
6. Ребро ещё не существует — идемпотентность: повторный вызов **не ошибка**, no-op или `DO NOTHING`

**Постусловия:**

- Вставка строки в `profile_follows` (если не было)  
- Счётчики обновлены (триггер или атомарно внутри RPC — **один** выбранный паттерн на весь проект; при триггере — только паттерн из §3.5, O(1) на две строки `profiles`)  
- Запись нотификации (если есть) — в теле RPC после вставки ребра, не в «толстом» триггере на `profile_follows` (§3.5.5)

**Коды ошибок (рекомендуется `raise ... USING errcode`):**

| Условие | Код / имя |
|--------|-----------|
| Не авторизован | `not_authenticated` |
| Цель не найдена | `user_not_found` |
| Подписка на себя | `cannot_follow_self` |
| Цель в sleep | `user_sleeping` |
| Заблокировано | `user_blocked` |
| Rate limit follow | `follow_rate_limited` |

### 4.2 `public.unfollow_user(p_target uuid) RETURNS void`

- Удаление ребра `follower_id = auth.uid()`, `following_id = p_target`  
- Идемпотентность: если ребра нет — no-op  
- Счётчики уменьшаются триггером / транзакционно

### 4.3 `public.is_following_user(p_target uuid) RETURNS boolean`

- Для UI; при `auth.uid()` null → `false`

### 4.4 `public.list_profile_followers(p_profile_id uuid, p_limit int, p_offset int)`

- Возвращает набор мини-профилей подписчиков (`profile_id`, `username`, `avatar_url`)  
- Учёт `reset_at` для отображения «noName» / пустой аватар — как в существующих enriched-RPC (единый стиль)

### 4.5 `public.list_profile_following(p_profile_id uuid, p_limit int, p_offset int)`

- Аналогично для «на кого подписан»

### 4.6 `public.list_following_feed(p_limit, p_cursor, …)` — **новый контракт ленты по подпискам**

**Вход:** пользователь = `auth.uid()`.

**Логика:**

1. Множество `following_id` из `profile_follows` где `follower_id = auth.uid()`  
2. Посты авторов из этого множества, удовлетворяющие **тем же правилам видимости**, что и публичная лента (не удалён, не архив, автор не скрыт по `account_state` / RLS)  
3. Сортировка: по умолчанию recency; поля для будущего ranking — зарезервировать в сигнатуре или вынести в v2

**Примечание:** текущий **hot feed** и **профильный feed** не заменяются; это **отдельный** продуктовый endpoint.

### 4.7 `public.wake_up_if_needed()` — уже в проекте

**Требование ТЗ:** вызывается при **установленной сессии** (см. клиентский контракт вне этого файла).  

**Поведение в БД:** атомарный `UPDATE profiles SET account_state = 'active', content_visible = true WHERE id = auth.uid() AND account_state = 'hibernate'` (или эквивалент), без гонок «два клиента».

### 4.8 `public.hibernate_account()` — уже в проекте

Не изменяет `profile_follows`. Ограничение «новые подписки на спящего» обеспечивается в `follow_user`.

## 5. Единая политика взаимодействия

### 5.1 `public.can_user_interact(a uuid, b uuid) RETURNS boolean` (или `text` с причиной отказа)

**Обязательное использование в:**

- `follow_user`  
- `list_following_feed` (фильтрация источников при необходимости)  
- дальнейшие: mentions, chat — вне скоупа миграции follow, но **тот же контракт**

**Минимальные проверки:**

- существование профилей  
- не self (для follow — отдельно дублируется)  
- нет блокировки по правилам `profile_blocks`  
- цель не «удалён / недоступен» (если введён флаг — см. п. 8.3)

## 6. Reconcile счётчиков и мониторинг

Дополняет §3.5: при **минимальных** триггерах дрейф редок, но reconcile обязателен для долгой жизни системы и для hot-row / миграционных краёв.

### 6.1 Задача

Периодически (например **daily**):

```text
Для каждого profiles.id (или батчами):
  true_followers = COUNT(*) FROM profile_follows WHERE following_id = id
  true_following = COUNT(*) FROM profile_follows WHERE follower_id = id
  Сравнить с profiles.followers_count / following_count
  При расхождении — UPDATE на true_* (или лог + ручной разбор)
```

### 6.2 Алерт

- Метрика: `abs(cached - true)` по пользователям или доля пользователей с drift  
- Порог — задать в инфраструктуре (Supabase Log Drains / внешний мониторинг)

### 6.3 Реализация на Supabase

- Предпочтительно: **`pg_cron`** + функция `reconcile_profile_follow_counts()` с `SECURITY DEFINER` и ограничением по батчам  
- Альтернатива: Edge Function по расписанию с `service_role` (менее предпочтительно для тяжёлых полных сканов)

## 7. Rate limiting

Два **независимых** класса лимитов:

| Класс | Пример | Где проверять |
|--------|--------|----------------|
| Social spam | 200 follow / час на `follower_id` | В начале `follow_user` (таблица событий / скользящее окно по `created_at` в `profile_follows` или отдельная `rate_limit_events`) |
| Account lifecycle | 30 дней между `hibernate` / (если включат) `reset` | Уже в логике `hibernate_account` / исторически `reset_account` |

**Требование:** не смешивать квоты в одну проверку.

## 8. Состояние аккаунта и reset (целевое vs текущее)

### 8.1 Hibernate (sleep)

- Граф **не меняется**  
- `follow_user` **отклоняет** подписку на цель в `hibernate`  
- Контент скрывается существующими RLS / RPC (не дублировать противоречивую логику)

### 8.2 Wake

- См. п. 4.7; клиент обязан вызывать после успешной сессии (отдельный контракт приложения)

### 8.3 Reset — разделение документации

| Уровень | Описание |
|--------|----------|
| **Target architecture** | Полный wipe контента; граф **не трогать**; счётчики графа не пересчитывать из wipe |
| **Current product** | `reset_account` **отключён** для клиента; основной сценарий — **hibernate** |

Любая новая миграция по reset должна явно ссылаться на этот раздел.

### 8.4 Удаление / soft-delete профиля

Зафиксировать продуктово:

- hard-delete профиля → CASCADE на рёбра графа **допустим** (нет сиротских рёбер)  
- soft-delete → либо запрет follow на «удалённый», либо каскадное удаление рёбер — **одно** правило на весь проект

## 9. RLS и доступ

### 9.1 `profile_follows`

Рекомендуемая модель для MVP:

- Прямой `INSERT`/`DELETE` с PostgREST **запрещён** для `authenticated`  
- Чтение списков — через RPC **или** узкие `SELECT` политики (если нужен PostgREST без RPC)

### 9.2 `profile_blocks`

- Пользователь видит только свои блокировки (политика по `blocker_id = auth.uid()`)
- Запись блокировки — отдельный RPC `block_user` / `unblock_user` (в скоупе отдельной задачи, но таблица — в этом ТЗ)

## 10. Лента: lazy-модель и клиент

**Supabase обязанность:** корректный ответ `list_following_feed` после появления ребра.

**Клиент (вне Postgres):** после успешного `follow_user` — **invalidate + reload** кэша ленты (модель A из продуктового ТЗ).

## 11. Нефункциональные требования

- **Производительность:** планы запросов для `list_following_feed` с индексами на `posts(user_id, created_at desc)` и фильтром по множеству following (батч / `= ANY(array)`).  
- **Безопасность:** все мутации графа только с проверкой `auth.uid()`; нет «доверенного» клиентского пути обхода.  
- **Миграции:** идемпотентные `IF NOT EXISTS` где уместно; новые RPC — `REVOKE ALL FROM public` + явные `GRANT`.  
- **Версионирование:** изменения контракта RPC фиксировать в `MIGRATIONS_INDEX.md` и в этом SPEC.

## 12. Приёмочные критерии (бэкенд)

1. Дважды вызванный `follow_user` на ту же цель не создаёт второе ребро и не ломает счётчики.  
2. `unfollow_user` без ребра — без ошибки, счётчики не уходят в минус.  
3. Подписка на пользователя в `hibernate` — отказ с кодом `user_sleeping`.  
4. Подписка при активной блокировке — отказ с кодом `user_blocked`.  
5. После вставки ребра счётчики у обоих профилей соответствуют `COUNT(*)` из `profile_follows`.  
6. `notification_events` (или аналог) не содержит двух записей с одним `dedupe_key` для одного follow-события.  
7. Reconcile при искусственно сломанном счётчике восстанавливает значение (тест на стейдже).  
8. `wake_up_if_needed` переводит только `hibernate → active` для текущего uid, идемпотентно.  
9. Триггерные функции на `profile_follows` не содержат `COUNT` по графу, внешних HTTP/notify и записи в `notification_events` (см. §3.5); при code review отклонять расширение тела без пересмотра производительности.
